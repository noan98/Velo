# Velo

Windows 向けの**軽量・高速なファイラー**（土台）。Rust + [Slint](https://slint.dev/) 製。

デザイン性よりも「起動の速さ・一覧表示の速さ・低メモリ」を最優先しています。
これは完成形ではなく、後から機能を足していける**堅牢な基盤**です。

---

## できること（現状のスコープ）

- 起動時にホームディレクトリの内容を一覧表示する
- 各行に **種別（📁 / 📄）・名前・サイズ・更新日時** を表示
- フォルダを**ダブルクリック**で中へ入る／**「↑ 上へ」**で親へ戻る
- 現在のパスをアドレスバー（テキスト）に表示
- ディレクトリ読み込みは**バックグラウンドスレッド**で行い、UI が固まらない
- 外部からのファイル追加 / 削除 / リネームを**監視**し、デバウンス後に一覧へ反映

## まだやらないこと（後回し・拡張ポイント）

- アイコン / サムネイル取得（今は種別記号で代用）
- コンテキストメニュー、コピー / 移動 / 削除などの操作
- 検索、ソート UI、タブ、複数ペイン、ブックマーク、設定画面

## ビルドと実行

ターゲットは **Windows (x86_64-pc-windows-msvc)** です。

```sh
cargo run              # 開発実行
cargo build --release  # リリースビルド（LTO で軽量化）
cargo clippy           # 静的チェック（警告ゼロを維持する）
```

> 注: 起動時は `dirs::home_dir()` を表示します。取得できない環境ではカレントディレクトリを使います。

## アーキテクチャ（ジュニア向けメモ）

設計の背骨は **「状態の真実の源（source of truth）は Rust 側に置き、Slint は表示専用にする」** です。
これを守ると、機能追加時に “UI とロジックがどちらの状態を信じればいいか分からない” という事故を防げます。

### 4 つのルール

1. **状態は Rust が持つ。** 「今どのディレクトリの、何を表示しているか」は
   [`AppState`](src/app_state.rs)（UI スレッド上の `thread_local`）が保持します。
   Slint 側のプロパティは、この状態を整形して映すだけです。

2. **重い I/O を UI スレッドで実行しない。** `read_dir` と各ファイルの `metadata` 取得は
   [`fs::lister::list_dir`](src/fs/lister.rs) にまとめ、必ず**別スレッド**で呼びます。

3. **スレッド → UI の反映は `upgrade_in_event_loop` 経由のみ。** ワーカースレッドは
   I/O 結果（`Send` なデータ）を返すだけで、UI のプロパティ/モデルを直接触りません。
   反映は [`main.rs`](src/main.rs) の `load_dir` → `apply_listing` で行います。

4. **一覧は仮想化 `ListView` + `VecModel` で供給する。** `ListView` は可視範囲の行だけを
   実体化するので 1 万件規模でも軽い。更新時は `VecModel::set_vec` で**モデルだけ差し替え**、
   プロパティ全体は作り直しません。

### データの流れ

```text
ユーザー操作（ダブルクリック / 上へ）
        │  (UI スレッド)
        ▼
   navigate_to ──→ current_dir を更新（最後の移動が勝つ）
        │
        ▼
   load_dir ──→ std::thread::spawn ─────────────┐  (ワーカースレッド)
                                                 ▼
                              fs::lister::list_dir で読み込み
                              to_rows で表示用文字列へ整形
                                                 │
                  weak.upgrade_in_event_loop ←───┘  (UI スレッドへ載せ替え)
                                                 ▼
                              apply_listing：モデル差し替え + パス更新
                              （ユーザー移動時はウォッチャを張り直す）
```

### ファイル監視

[`fs::watcher`](src/fs/watcher.rs) が `notify` + `notify-debouncer-full` で
カレントディレクトリを**非再帰**監視します。生イベントは 1 操作で複数飛ぶため
300ms デバウンスし、確定したら「何か変わった」として一覧を再読み込みします。
監視コールバックは別スレッドで呼ばれるので、ここでも `upgrade_in_event_loop` で
UI スレッドに載せ替えます。

### 「最後の移動が勝つ」ガード

フォルダを連打したときに古い読み込み結果が後から届くと、表示がずれます。
これを防ぐため `navigate_to` で `current_dir` を即座に更新し、`apply_listing` では
`current_dir == 読み込んだパス` のときだけ反映します（違えば破棄）。

## モジュール構成

```text
src/
  main.rs        エントリ。UI とバックエンドの配線、スレッド↔UI の橋渡し
  app_state.rs   アプリ状態（カレントパス・現在のエントリ・ウォッチャ）
  fs/
    mod.rs
    entry.rs     FileEntry：ドメインモデル（整形なしの生データ）
    lister.rs    別スレッドで read_dir → Vec<FileEntry>
    watcher.rs   notify + debouncer。変更をコールバックで通知
ui/
  app.slint      メインウィンドウ + 仮想化一覧
build.rs         slint-build で .slint をコンパイル
```

## 将来の拡張ポイント

- **アイコン / サムネイル:** 今は `FileRow.type-label` に記号を入れているだけ。
  将来は「バックグラウンド取得 + LRU キャッシュ」でパスごとの画像を後追い供給できるよう、
  ドメインの `FileEntry` と表示用の `FileRow` を分離してあります。
- **ソート / フィルタ:** `FileEntry` が生の値（`u64` サイズ・`SystemTime`）を保持するため、
  整形前の値でソート/フィルタを実装できます。
- **遅延読み込み:** モデル境界（`VecModel<FileRow>`）を分離してあるので、
  将来 `Model` トレイトを自前実装して遅延供給に差し替えられます。

> 方針: **まず動く土台を最優先。最適化は必ず計測してから行うこと。**
